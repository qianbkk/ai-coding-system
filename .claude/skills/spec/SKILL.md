# Skill：spec — 设计规格文档

**触发条件：** 复杂路径任务，clarify 完成后；或任何改动 ≥ 4 文件的任务
**前置要求：** 有清晰的意图声明（来自 clarify skill 或用户直接提供）
**下一步：** plan skill
**输出文件：** `docs/specs/YYYY-MM-DD-<topic>.md`

---

## 目标

把意图声明转化为可执行的设计合同。这份文档：
- 是实现阶段的参照标准
- 是 review 阶段的验收依据
- 是未来回看时的决策记录

---

## Spec 文档结构

按以下结构生成，分段呈现，每段后等用户确认再继续：

---

### 第 1 段：概述（先呈现，等确认）

```markdown
# [功能/任务名称]
_日期：YYYY-MM-DD_

## 一句话定义
[用一句话说清楚：这是什么、为了谁、解决什么问题]

## 背景与动机
[为什么现在做这个，不做会怎样]
```

---

### 第 2 段：范围（确认第 1 段后呈现）

```markdown
## 范围

### 在范围内
- [功能 1]
- [功能 2]

### 明确排除（重要）
- [不做的事 1]——原因：[为什么排除]
- [不做的事 2]——原因：[为什么排除]

### 约束条件
- 技术约束：[例：必须兼容现有 API]
- 性能约束：[例：P95 响应时间 < 200ms]
- 安全约束：[例：需要 auth 验证]
```

---

### 第 3 段：技术设计（确认第 2 段后呈现）

```markdown
## 技术设计

### 架构概述
[用 1-3 段话描述整体架构，避免无关细节]

### 主要组件
| 组件 | 职责 | 位置 |
|------|------|------|
| [组件名] | [做什么] | [文件路径] |

### 数据流
[描述数据从哪来、经过哪些处理、到哪去]

### 接口定义（如有）
[API 端点、函数签名、事件格式等]

### 关键决策
| 决策 | 选择 | 原因 | 放弃的替代方案 |
|------|------|------|--------------|
| [决策点] | [选择] | [为什么] | [其他选项 + 放弃原因] |
```

---

### 第 4 段：验收标准（确认第 3 段后呈现）

```markdown
## 验收标准

### 功能验收
- [ ] [用户故事格式：当 X 时，应该 Y]
- [ ] [边界情况：当 X 时，应该 Y]

### 非功能验收
- [ ] 性能：[具体指标]
- [ ] 安全：[具体要求]
- [ ] 错误处理：[失败场景的期望行为]

### 不在验收范围内
- [明确说明哪些问题不属于本次任务]
```

---

## Spec 自检（输出前在内部完成）

```
□ 有没有占位符没填（[TODO]、[TBD]）？
□ 有没有自相矛盾的要求？
□ 范围是否清晰？有没有模糊的"也许"？
□ 验收标准是否可测试？（"好用"不可测，"响应 < 200ms"可测）
□ 有没有遗漏的边界情况？
```

发现问题先修复，再提交给用户。

---

## Spec 保存

```bash
# 保存位置
docs/specs/YYYY-MM-DD-<topic-kebab-case>.md

# 提交
git add docs/specs/
git commit -m "spec: add [topic] design document"
```

保存并提交后，告知用户文档位置，等待确认进入 plan 阶段。
