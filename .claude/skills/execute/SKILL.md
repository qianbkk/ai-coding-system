# Skill：execute — 实现

**触发条件：** 有明确任务（来自 plan 或直接给出），准备写代码
**适用路径：** 简单路径 + 复杂路径（逐任务调用）
**下一步：** 复杂路径 → 下一个任务 或 review | 简单路径 → 完成

---

## 开始前（每次必做，30 秒）

在内部回答，有不确定的提问：

```
① 这个任务的范围是什么？（改哪些文件，不改哪些文件）
② 完成的验证方式是什么？（跑哪个测试？执行哪个命令？）
③ 有没有安全影响？（Auth、用户输入、密钥、权限）
```

安全相关的任务，在开始前告知用户："这涉及 [Auth/权限/密钥]，我会实现，但完成后建议你做安全 review。"

---

## 实现决策：懒惰阶梯（每次实现决策前，按顺序检查）

```
第 1 级：删掉它 ──── 这个功能真的需要吗？不做会怎样？
第 2 级：标准库 ──── 语言/框架内置能做？不安装额外依赖
第 3 级：原生 API ── 浏览器/运行时原生能做？不引入 polyfill
第 4 级：已有代码 ── 项目里已有实现？搜索后复用
第 5 级：小依赖 ──── 一个轻量专注的包能做？检查体积和维护状态
第 6 级：自己写 ──── 以上都不行，写最小可行实现
```

**选了 3-6 级时，在代码旁边打注释：**
```
// shortcut: [选择的方案及原因]
// 上限: 当 [条件] 时需要升级
// 升级路径: [更完整的方案或参考位置]
```

## TDD 协议（有测试框架的项目强制执行）

```
Step 1：写 failing test
  → 明确描述期望行为
  → 运行测试，确认它失败（不是"应该会失败"，是实际运行）
  → git commit: "test: [描述]（failing）"

Step 2：写最小实现
  → 只写让测试通过所需的代码，不多写
  → 运行测试，确认通过

Step 3：重构
  → 清理重复，改善命名，保持测试绿色
  → git commit: "feat: [描述]"
```

**没有测试框架时：** 在完成后提供手动验证步骤（可以执行的命令或操作序列）。

## 代码质量规则（写代码时遵守）

**结构**
- 每个函数只做一件事；用"和"描述它做什么时，说明需要拆
- 函数长度超过 30 行时，先判断是否可以拆分
- 不写"以后可能用到"的参数、配置项或抽象层

**命名**
- 命名说意图，不描述实现：`fetchUserById` 而不是 `getData`
- 布尔变量用 `is/has/can/should` 前缀
- 不缩写（除非领域通用）：`usr → user`，`btn → button`

**错误处理**
- 所有 async 函数必须有错误处理路径
- 错误信息必须包含可调试的上下文（ID、操作名、来源）
- 不写空 catch 块；必须写时，注释说明为什么可以忽略

**注释**
- 注释写"为什么"，不写"是什么"（代码本身说"是什么"）
- 非直觉的决策、已知的 workaround、绕过规则的原因必须注释
- 不写描述代码行为的注释：`// 给 i 加 1`

**引用规则**
对于框架 API、库的使用方式，使用官方文档作为依据。不确定时，明确说出来：
```
"我不确定 [X] 在 [版本] 里的行为，建议确认官方文档后再决定。"
```

## 完成前自检（说"完成"之前，内部过一遍）

**范围**
- [ ] 有没有改范围外的代码？（包括"顺手优化"）
- [ ] 改动是否都能溯源到当前任务？

**正确性**
- [ ] 测试实际运行了（不是"应该会过"）
- [ ] 覆盖了主流程和关键边界情况
- [ ] 有错误处理，失败路径有期望行为

**安全**
- [ ] 没有硬编码密钥、token、密码
- [ ] 用户输入经过验证（类型、范围、格式）
- [ ] 没有 SQL 拼接或路径穿越风险

**代码质量**
- [ ] 没有未使用的 import、变量、函数
- [ ] 选了 shortcut 的地方有注释标注上限和升级路径
- [ ] 非直觉的代码有注释说明原因

**发现任何"否"，先修复再说完成。**
