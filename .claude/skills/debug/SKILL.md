# Skill：debug — 结构化调试

**触发条件：** 遇到 bug、测试失败、行为不符合预期、错误信息不明确
**适用路径：** 简单路径 + 复杂路径均适用
**下一步：** 找到根因后 → execute（修复）

---

## 原则

**不要在没有理解根因的情况下修复 bug。**

猜测 + 随机修改 = 覆盖了症状，保留了原因，制造了新的 bug。

---

## 六步调试协议

### Step 1：复现（Reproduce）

先稳定复现，再分析原因。

```
□ 能稳定复现吗？（每次都发生，还是偶发？）
□ 最小复现步骤是什么？
□ 在什么环境下发生？（本地/CI/生产/特定浏览器）
□ 什么时候开始出现的？（最近的 commit？）
```

**不能稳定复现时：** 先找到让它稳定发生的条件，再继续。偶发 bug 最危险。

---

### Step 2：最小化（Minimize）

缩小问题范围，找到最小的触发条件。

```
□ 移除不相关的代码/数据，问题还在吗？
□ 能写一个最小化的复现用例吗？（10 行以内的测试）
□ 问题在哪个层级？（UI / API / 业务逻辑 / 数据层）
```

目标：写一个最小的 failing test，精确描述期望行为和实际行为的差异。

---

### Step 3：假设（Hypothesize）

在查看代码之前，先提出假设。

输出格式：
```
当前现象：[描述观察到的行为]
期望行为：[应该发生什么]
差异：[两者的具体区别]

可能的原因（按可能性排序）：
1. [假设 A]——如果是这个，会有什么证据？
2. [假设 B]——如果是这个，会有什么证据？
3. [假设 C]——如果是这个，会有什么证据？
```

不要跳过这步直接看代码——先有假设，再用证据验证或推翻。

---

### Step 4：插桩（Instrument）

根据假设，有针对性地添加调试信息。

```
□ 在假设的关键路径上添加日志/断点
□ 打印中间状态，而不是最终结果
□ 验证假设 A：[具体操作] → 结果？
□ 如果假设 A 不成立，验证假设 B：[具体操作] → 结果？
```

**记录每个假设的验证结果。** 不要保留临时调试代码；找到根因后记得清理。

---

### Step 5：修复（Fix）

找到根因后：

```
□ 理解了为什么会发生吗？（不是"怎么修"，是"为什么出错"）
□ 修复是针对根因的，还是只处理了症状？
□ 这个修复会影响其他地方吗？
```

修复原则：
- 修复根因，不是掩盖症状
- 修复应该最小化（不要在修 bug 时重构无关代码）
- 如果正确修复需要较大改动，先问用户

---

### Step 6：回归测试（Regression）

```
□ 为这个 bug 写一个测试（如果之前没有）
□ 确认测试先失败（复现 bug）
□ 确认修复后测试通过
□ 运行完整测试套件，确认没有引入新问题
```

**为什么要加回归测试：** 因为这个 bug 已经发生过一次，说明原有测试没有覆盖这个场景。加测试就是防止它再次发生。

---

## 完成后的知识提取

```
根因分析：
  这个 bug 是由 [根因] 引起的
  之所以没有被测试覆盖，是因为 [原因]
  
预防措施：
  □ 加了回归测试覆盖这个场景
  □ 建议加到 CLAUDE.md Gotcha 列表（如果是项目特有的陷阱）
  □ 相关文档/注释是否需要更新？
```

---

## 当调试卡住时

如果用了超过 15 分钟没有进展：

1. 重新描述当前的现象和已知的事实（不是假设）
2. 列出已经排除的假设及原因
3. 告诉我你卡在哪里
4. 我来决定是继续调试还是换策略（bisect / 新 session / 寻求其他资源）

**不要在同一个方向上无限尝试。** 卡住 15 分钟 = 需要切换策略。
