# Skill：plan — 任务拆解与排序

**触发条件：** spec 完成后；或用户要求拆分一个大任务
**前置要求：** spec 文档或清晰的需求描述
**下一步：** execute skill（逐任务）
**输出文件：** `docs/plans/YYYY-MM-DD-<topic>-tasks.md`

---

## 目标

把 spec 文档转化为：
- 有顺序的任务列表（依赖关系正确）
- 每个任务的大小适合在一次对话里完成
- 每个任务有明确的验收标准

---

## 任务拆分规则

### 大小限制（每个任务必须满足）
```
□ 改动文件 ≤ 5 个
□ 预计实现时间 ≤ 30 分钟
□ 有独立的验收标准（不依赖其他未完成的任务才能验证）
□ 改动逻辑内聚（改的东西属于同一个关注点）
```

如果一个任务不满足以上条件，继续拆分。

### 依赖排序规则
```
基础设施 > 数据层 > 业务逻辑 > API 层 > UI 层 > 测试 > 文档
```

- 标注 `[阻塞]`：此任务完成前，后续某任务无法开始
- 标注 `[并行]`：此任务可以与其他任务同时进行
- 无标注：按列表顺序执行即可

---

## 任务列表格式

```markdown
# 任务列表：[功能名称]
_基于 Spec：docs/specs/YYYY-MM-DD-<topic>.md_
_日期：YYYY-MM-DD_

## 任务

### T01 · [任务名称] [阻塞 T03]
**目标：** [一句话说明这个任务做什么]
**改动范围：**
- `src/xxx.ts` — [做什么改动]
- `src/yyy.ts` — [做什么改动]
**验收标准：**
- [ ] [可验证的完成条件]
- [ ] [边界情况]
**状态：** 待开始

---

### T02 · [任务名称] [并行]
...

---

### T03 · [任务名称]（依赖 T01）
...
```

---

## 拆分质量检查

输出任务列表前，在内部验证：

```
□ 每个任务的验收标准都是可测试的（不是"感觉好用"）
□ 依赖关系有没有循环（A 依赖 B，B 依赖 A）？
□ 第一个任务是否可以立即开始（不依赖任何未完成的东西）？
□ 任务之间的边界是否清晰（没有"T02 也需要改 T01 里的文件"）？
□ 有没有遗漏的任务（spec 里有但 plan 里没有对应的任务）？
```

---

## 呈现方式

先输出任务总览（任务名 + 预计时间 + 依赖关系图），等用户确认后再输出完整任务列表。

```
任务总览：
T01 [任务名] (15min) 
  ↓ 阻塞
T02 [任务名] (20min)     T03 [任务名] (10min) [并行]
  ↓
T04 [任务名] (25min)
...

共 X 个任务，预计总时间：Y 分钟
```

用户确认总览后，保存完整任务列表到文件，然后从 T01 开始执行。

---

## 执行时的进度跟踪

每完成一个任务，在任务列表里更新状态：
- `待开始` → `进行中` → `已完成` / `已跳过（原因）`

如果一个任务在执行中发现需要拆分，暂停并告知用户，重新拆分后再继续。
