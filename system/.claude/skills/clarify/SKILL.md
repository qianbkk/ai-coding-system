# Skill：clarify — 需求澄清

**触发条件：** 需求模糊、目标不明确、存在多种解读、用户说"帮我想想"或"我有个想法"
**适用路径：** 简单路径（可选）+ 复杂路径（推荐）
**下一步：** 简单任务 → execute | 复杂任务 → spec

---

## 目标

在写任何代码之前，搞清楚两件事：
1. **真实意图**：用户真正想要什么（不只是字面描述）
2. **成功标准**：什么状态算"做到了"

---

## Phase 0：静默分析（执行，不输出）

收到初始描述后，先在内部完成：

```
□ 识别领域（软件功能 / 架构 / 调试 / 学习 / 其他）
□ 列出隐含假设（用户没说但你假设的事情）
□ 找出最关键的未知维度（最影响方向的那个）
□ 判断复杂度（简单 / 中等 / 复杂）
```

不要把这个分析输出给用户。直接进入 Phase 1。

---

## Phase 1：第一个问题

**规则：每次只问一个问题，不叠问。**

从以下维度选一个最关键的问，优先级从高到低：

1. **目标/动机** — 这要解决什么问题？为什么现在要做？
2. **使用场景** — 谁用这个？在什么情况下？
3. **约束** — 有什么不能做的？时间/技术/团队限制？
4. **成功标准** — 什么状态算完成？怎么判断好坏？
5. **已有条件** — 已经有什么了？试过什么？

问题形式优先用选择题：
```
"A 还是 B？还是两者都有？"
```

不要一次问多个问题，也不要把问题埋在一大段解释里。

---

## Phase 2：迭代追问

每轮回答后：

1. 更新内部的假设清单（确认了什么？推翻了什么？）
2. 判断当前最模糊的维度，问下一个问题
3. 如有重要的"用户可能没想到的问题"，主动点出：

```
"有一个你可能还没考虑到但会影响方向的问题：[X]。
这会影响 [Y]，你有想法吗？"
```

**清晰度自检（内部，每轮后）：**

| 维度 | 权重 | 清晰？ |
|------|------|--------|
| 核心目标 | ★★★ | |
| 使用场景/受众 | ★★★ | |
| 关键约束 | ★★ | |
| 成功标准 | ★★ | |
| 已有条件 | ★ | |

**推进条件：** ★★★ 维度全部清晰，且 ★★ 维度至少一个清晰。

---

## Phase 3：提出方向（满足推进条件后）

提出 2-3 个不同的实现方向，每个方向：
- 一句话描述
- 适合的场景
- 主要代价/取舍

然后说明你倾向哪个及原因，等用户确认。

**方向必须有真实差异（路径差异，不是细节差异）。**

---

## Phase 4：输出意图声明

用户确认方向后，输出：

```markdown
## 任务意图声明

**要做的：** [一句话]
**不做的：** [明确排除的内容]
**选定方向：** [方向名称 + 核心逻辑]
**成功标准：** [可验证的完成条件]
**关键假设：** [需要用户确认的前提]

---
下一步：[spec（复杂）/ execute（简单）]
```

等用户确认这个意图声明后，才进入下一步。
