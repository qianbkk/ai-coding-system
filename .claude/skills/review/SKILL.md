# Skill：review — 代码审查

**触发条件：** 复杂路径任务全部完成后；或任何 PR 合并前；或用户要求 review
**前置要求：** 有可以 review 的代码改动
**下一步：** 问题已修复 → 告知用户可以合并 | 有重大问题 → 返回 execute

---

## 目标

从多个视角检查代码，找到在"感觉完成了"的状态下容易遗漏的问题。

**原则：review 是找问题的，不是找借口放行的。**

---

## 三个视角（依次进行）

### 视角一：质量工程师

关注：代码是否正确、可维护、没有隐患

```
□ 主流程是否正确？有没有逻辑错误？
□ 边界情况是否处理了？（空值、空列表、最大值、并发）
□ 错误路径是否有处理？失败时的行为是否可预期？
□ 有没有重复代码可以提取？
□ 有没有不必要的复杂度？（能用 10 行做到的不用 50 行）
□ 测试是否覆盖了关键场景？测试是测行为还是测实现细节？
```

### 视角二：安全工程师

关注：代码是否有安全漏洞

```
□ 有没有硬编码的密钥、token、密码？（包括注释里的）
□ 用户输入是否经过了验证？（类型、范围、格式、长度）
□ 有没有 SQL 拼接？（应该用参数化查询）
□ 有没有路径穿越风险？（文件操作需要路径规范化）
□ 错误信息是否暴露了内部实现？（堆栈跟踪、数据库结构）
□ 有没有不安全的默认值？（CORS 全开、无 rate limiting）
□ Auth 相关的代码是否有正确的权限检查？
```

### 视角三：可维护性工程师

关注：6 个月后，另一个人能否快速理解和修改这段代码

```
□ 命名是否说明了意图？（看名字能猜到它做什么）
□ 非直觉的决策是否有注释说明原因？
□ 有没有 shortcut 没有注释上限和升级路径？
□ 公共 API 是否有文档注释？
□ 依赖关系是否清晰？（模块间的耦合程度）
□ 有没有"魔法数字"没有命名为常量？
```

---

## 反合理化检查（对发现的问题执行）

当你发现问题但"感觉可以接受"时，用以下方式验证：

| 你的想法 | 反驳它 |
|---------|--------|
| "这只是原型，之后再加测试" | 原型往往变成生产代码，债务会累积 |
| "这个 case 很少发生，不处理也行" | 最罕见的 case 往往是最危险的 bug |
| "这段代码我看得懂，不用注释" | 三个月后的你，或者你的协作者，不一定看得懂 |
| "性能还好，不需要优化" | 要有具体数据，不能凭感觉 |
| "auth 这里应该没问题" | Auth 相关的必须明确验证，"应该没问题"不够 |

## Review 输出格式

```markdown
## Code Review 结果

### 必须修复（Blocker）
这些问题在合并前必须解决：
- [问题描述] — [位置] — [建议修复方式]

### 建议修复（Non-blocker）
这些问题不阻止合并，但建议处理：
- [问题描述] — [位置] — [为什么建议修复]

### 安全注意（需要人工确认）
这些需要你自己判断或找专业人员审查：
- [问题描述] — [风险说明]

### 通过
这些方面检查通过：
- 功能正确性：✅
- 错误处理：✅ / ⚠️ [说明]
- 安全基础：✅ / ⚠️ [说明]
- 代码质量：✅ / ⚠️ [说明]

### 总体评估
[可以合并 / 修复 Blocker 后合并 / 需要较大修改]
```

---

## 发现 Blocker 时

不要"顺手"修复，而是：
1. 列出所有 Blocker
2. 等用户决定优先级
3. 用户确认后，返回 execute skill 修复

修复完成后，对修复的部分重新 review，不需要全量重新 review。
